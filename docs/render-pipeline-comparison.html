Render Pipeline · Aperture's 6 phases vs three.js & PlayCanvas (WebGPU)

Aperture organizes its render pipeline into six explicit, named phases — extract → asset-change-collection → prepare → queue → sort → submit — each with a typed input/output contract and pooled scratch state. This document maps those six phases against the equivalent work in three.js's WebGPURenderer and PlayCanvas's WebGPU graphics backend, showing where the other engines bundle, interleave, or skip the same conceptual stages.

interface RenderSnapshot {
  frame: number;
  views:           ViewPacket[];
  meshDraws:       MeshDrawPacket[];   // per-submesh renderables w/ sortKey + batchKey
  lights:          LightPacket[];
  environments:    EnvironmentPacket[];
  shadowRequests:  ShadowRequestPacket[];
  bounds:          BoundsPacket[];
  transforms:      Float32Array;        // packed mat4s, 16 floats each
  viewMatrices:    Float32Array;        // packed view + projection
  diagnostics:     RenderDiagnostic[];
  report:          RenderSnapshotReport;
}

SUB-STEPS

  1. Register Camera / Light / Mesh / Material authoring components with ECS query manager
  2. extractViews — query [Camera]; validate WorldTransform; compose view/proj/viewProj matrices; populate ViewPacket
  3. extractLights — query [Light]; classify (ambient/directional/point/spot/environment); emit LightPacket, EnvironmentPacket, ShadowRequestPacket
  4. extractMeshDraws — query [Mesh, Material]; run 11 validation gates; for each submesh emit MeshDrawPacket with computed renderId, sortKey, batchKey
  5. Push every world matrix into the snapshot's packed transforms: Float32Array
  6. Sort meshDraws by initial sortKey (deterministic order before phase 5)

WHAT HAPPENS BEFORE THE WALK

  1. Update camera coordinate system / depth reversal flag (WebGPU vs WebGL)
  2. Fetch or build RenderContext for this (camera, renderTarget) pair — viewport, scissor, clear values, depth/stencil state
  3. Fetch or build RenderList for this (scene, camera) pair (cached across frames)

No serializable snapshot exists; the renderer holds live refs to Scene, Camera, and every Mesh/Geometry/Material for the rest of the frame.

NO DIRECT EQUIVALENT. PlayCanvas reads live LayerComposition & MeshInstance refs each frame. The closest analogue is the renderer.update(comp) step which culls but doesn't snapshot.

WHAT HAPPENS BEFORE THE WALK

  1. root.syncHierarchy() — propagate transforms top-down
  2. _batcher.updateAll() — refresh dynamic batches
  3. renderer.update(comp) → cullComposition(comp) walks layers, frustum-culls per camera, and writes into CulledInstances { opaque, transparent } on the layer

The culled instance lists are mutations on live layer state, not copies. Cleared and rewritten each frame.

★ Where Aperture's Phase Model Pulls Away From The Others

1.  Explicit, named phases with typed I/O

APERTURESix phases declared in code (RENDER_FRAME_PHASE_ORDER), each with a RenderFramePhaseDescriptor stating what it consumes and produces. The pipeline is data, not just a call sequence.

THREE.JSPhases are call-site conventions inside _renderScene() — not addressable from outside the renderer. You can't ask “what's the queue look like right now?”

PLAYCANVASFrameGraph + RenderPass abstractions exist, but the inner work (prepare / queue / sort) is bundled inside renderForwardPrepareMaterials + renderForwardInternal.

2.  Snapshot is the worker boundary

APERTURERenderSnapshot is typed and structured-clone-friendly — phases 2–5 can run on the snapshot, which means they can run on a worker without the renderer ever touching live ECS state.

THREE.JSLives on live JS object refs (Mesh, Geometry, Material). Renderer must run on the thread that owns the scene.

PLAYCANVASSame — culling and queue building mutate live Layer state. Workerizing the renderer would require shipping the scene graph.

3.  Inspectable command stream before submit

APERTUREwriteRenderPassCommands emits a typed JS command list into pooled scratch before assembleFrameBoundary drives the encoder. You can log, diff, count, or replay the stream — it's data.

THREE.JSPer-draw, WebGPUBackend._draw calls pass.setPipeline / setBindGroup / drawIndexed directly. No intermediate list — only side-effects on the encoder.

PLAYCANVASSame as three.js for emission — device.draw() hits passEncoder directly inside renderForwardInternal().

4.  Zero-alloc scratch on the hot path

APERTUREEvery per-frame producer writes into a named scratch (RenderQueueScratch.recordPool, RenderPassCommandScratch.commandPool, transform-pack scratch, …). Steady-state frames don't allocate JS objects.

THREE.JSPer-frame object literals in render items, sort comparators, frustum math; WeakMap caches grow with scene. GC pressure visible at scale.

PLAYCANVAS_drawCallList is rebuilt per layer; RenderAction objects allocated per camera-layer-transparency combo per frame.
